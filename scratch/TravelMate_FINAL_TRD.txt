## 1. Executive Summary
This Technical Requirements Document (TRD) translates the TravelMate Product Requirements Document (PRD v2.0) into a comprehensive technical specification for implementation. The system is a trust-first group travel matching platform with safety as an architectural constraint.
**Core Technical Decisions:**
| Decision | Choice | Rationale |
|----------|--------|-----------|
| **Architecture** | Microservices + Event-Driven | Decoupling, independent scaling, auditability |
| **Primary Database** | PostgreSQL 16 | ACID compliance, JSONB support, partitioning capabilities |
| **Cache** | Redis 7 | Session management, matching result caching, distributed locking |
| **Event Bus** | Kafka 3.x | High-throughput, durable logs, exactly-once semantics |
| **API Layer** | Node.js 22 LTS | High I/O performance, built-in WebSocket support |
| **Internal RPC** | gRPC | High performance, streaming support, smaller payloads |
| **Orchestration** | Kubernetes 1.31 + EKS | Industry standard, auto-scaling, multi-region support |
| **Observability** | OpenTelemetry + Prometheus + Grafana + Jaeger | Standard CNCF stack, production monitoring |

## 3. Matching Algorithm — Detailed Specification
### 3.1 Multi-Stage Pipeline
```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         MATCHING PIPELINE                                  │
├
─────────────────────────────────────────────────────────────────────────────
┤
│                                                                             │
│  [1] DESTINATION FILTER                                                     │
│      └── Exact match required                                              │
│      └── SQL: WHERE destination = $1                                       │
│                                                                             │
│  [2] DATE WINDOW FILTER                                                     │
│      └── Minimum 2-day overlap required                                    │
│      └── SQL: WHERE start_date &lt;= $2 AND end_date &gt;= $1                    │
│                                                                             │
│  [3] BUDGET FILTER                                                         │
│      └── Tier match required (low/mid/high)                                │
│      └── SQL: WHERE budget_tier = $3                                       │
│                                                                             │
│  [4] GENDER PREFERENCE FILTER                                              │
│      └── Hard rejection on incompatibility                                 │
│      └── Logic:                                                             │
│          IF user_a.gender_preference = 'women-only' AND                     │
│             user_b.gender != 'F' THEN reject                               │
│          IF user_a.gender_preference = 'men-only' AND                       │
│             user_b.gender != 'M' THEN reject                               │
│                                                                             │
│  [5] AGE BUCKET PREFERENCE                                                 │
│      └── Soft preference; can be relaxed                                   │
│      └── Buckets: 18-24, 25-34, 35-44, 45+                                 │
│                                                                             │
│  [6] INTEREST EMBEDDING                                                    │
│      └── Convert tags to embedding vectors using pre-trained model         │
│      └── Tags: trek, party, photography, food, adventure, culture          │
│                                                                             │
│  [7] CANDIDATE POOL GENERATION                                             │
│      └── All users passing hard filters (steps 1-4)                        │
│      └── Max pool size: 500 users for performance                          │
│                                                                             │
│  [8] COMPATIBILITY GRAPH CONSTRUCTION                                      │
│      └── Weighted edges between candidates                                 │
│      └── Edge weight = compatibility_score(A, B)                           │
│      └── Store in UserCompatibility table for caching                      │
│                                                                             │
│  [9] CONSTRAINED GROUP OPTIMIZER                                           │
│      └── Greedy packing with local swap optimization                       │
│      └── Algorithm: see Section 3.3                                        │
│                                                                             │
│  [10] GROUP FORMATION                                                      │
│      └── Groups of 4-8 users with max compatibility                        │
│      └── Create Group and GroupMember records                              │
│      └── Emit GroupFormed event                                            │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```
### 3.2 Compatibility Score Formula (Implementation)
```python
def calculate_compatibility(user_a, user_b):
    """
    Calculate pairwise compatibility score between two users.
    Returns: float between 0 and 1
    """
    
    # Date Overlap (0.30 weight)
    date_overlap = max(0, min(user_a.end_date, user_b.end_date) - 
                           max(user_a.start_date, user_b.start_date))
    total_days = max(user_a.end_date - user_a.start_date,
                     user_b.end_date - user_b.start_date)
    date_score = min(1.0, date_overlap / total_days) if total_days &gt; 0 else 0
    
    # Interest Similarity (0.20 weight) - Jaccard similarity
    interests_a = set(user_a.interests)
    interests_b = set(user_b.interests)
    interest_score = len(interests_a &amp; interests_b) / len(interests_a | interests_b) if interests_a | interests_b else 0
    
    # Budget Match (0.15 weight)
    budget_tiers = {'low': 0, 'mid': 1, 'high': 2}
    budget_diff = abs(budget_tiers[user_a.budget_tier] - budget_tiers[user_b.budget_tier])
    budget_score = 1.0 if budget_diff == 0 else 0.5 if budget_diff == 1 else 0
    
    # Travel Style Compatibility (0.10 weight)
    style_score = cosine_similarity(user_a.travel_styles_vector, 
                                   user_b.travel_styles_vector)
    
    # Age Compatibility (0.10 weight)
    max_age_diff = 30  # Max reasonable age difference
    age_diff = abs(user_a.age - user_b.age)
    age_score = 1 - (age_diff / max_age_diff)
    
    # Language Compatibility (0.05 weight)
    lang_a = set(user_a.languages)
    lang_b = set(user_b.languages)
    lang_score = len(lang_a &amp; lang_b) / len(lang_a | lang_b) if lang_a | lang_b else 1.0
    
    # Previous Ratings (0.05 weight)
    rating_score = (user_a.avg_rating + user_b.avg_rating) / 2  # Normalized 0-1
    
    # Trust Score (0.05 weight)
    trust_score = (user_a.trust_score + user_b.trust_score) / 2
    
    # Weighted sum
    score = (0.30 * date_score + 
             0.20 * interest_score + 
             0.15 * budget_score + 
             0.10 * style_score + 
             0.10 * age_score + 
             0.05 * lang_score + 
             0.05 * rating_score + 
             0.05 * trust_score)
    
    return score
```
### 3.3 Group Formation Algorithm — Constrained Greedy with Local Swap
```python
def form_groups(candidates, min_size=4, max_size=8):
    """
    Form optimal groups from candidate pool using greedy packing with local swap.
    Returns: List of groups, each containing list of user IDs.
    """
    if len(candidates) &lt; min_size:
        return []  # Not enough candidates
    
    # Step 1: Sort candidates by "compatibility potential"
    # Higher potential = more compatible with others
    candidates_sorted = sorted(candidates, key=lambda u: u.compatibility_potential, reverse=True)
    
    groups = []
    remaining = set([u.id for u in candidates_sorted])
    
    # Step 2: Greedy packing
    while len(remaining) &gt;= min_size:
        # Start with the candidate with highest potential
        seed_id = max(remaining, key=lambda uid: get_compatibility_potential(uid, remaining))
        remaining.remove(seed_id)
        
        # Build group around seed
        group = [seed_id]
        
        # Add candidates based on compatibility with current group
        while len(group) &lt; max_size and remaining:
            # Find best candidate to add
            best_id = None
            best_score = -1
            
            for uid in remaining:
                # Average compatibility with current group
                avg_score = sum(get_compatibility_score(uid, gid) for gid in group) / len(group)
                if avg_score &gt; best_score:
                    best_score = avg_score
                    best_id = uid
            
            # Add if compatibility is reasonable
            if best_score &gt;= 0.3:  # Threshold
                group.append(best_id)
                remaining.remove(best_id)
            else:
                break
        
        # Ensure minimum group size
        if len(group) &gt;= min_size:
            groups.append(group)
        else:
            # Return unused candidates back
            remaining.update(group)
            break
    
    # Step 3: Local optimization - try swapping to improve compatibility
    for _ in range(5):  # Iteration limit
        improved = False
        for i, group_i in enumerate(groups):
            for j, group_j in enumerate(groups):
                if i &gt;= j:
                    continue
                # Try swapping one member between groups
                for uid_i in group_i:
                    for uid_j in group_j:
                        # Check if swap improves total compatibility
                        old_score = get_group_compatibility(group_i) + get_group_compatibility(group_j)
                        new_group_i = [x for x in group_i if x != uid_i] + [uid_j]
                        new_group_j = [x for x in group_j if x != uid_j] + [uid_i]
                        new_score = get_group_compatibility(new_group_i) + get_group_compatibility(new_group_j)
                        
                        if new_score &gt; old_score * 1.05:  # 5% improvement threshold
                            # Perform swap
                            group_i, group_j = new_group_i, new_group_j
                            improved = True
                            break
                    if improved:
                        break
                if improved:
                    break
            if improved:
                break
        if not improved:
            break
    
    return groups
```
### 3.4 Complexity Analysis
| Stage | Time Complexity | Space Complexity |
|-------|----------------|------------------|
| Filtering | O(n) | O(n) |
| Compatibility Score Computation | O(k²) where k = candidates after filtering | O(k²) for caching |
| Greedy Packing | O(k × max_size × avg_compatibility) | O(k) |
| Local Swap Optimization | O(g² × avg_size²) | O(1) |
| **Total** | **O(n + k² + g² × avg_size²)** | **O(k²)** |
Where:
- n = total users
- k = candidates after filtering (typically ≤ 500)
- g = number of groups formed
- avg_size = average group size (4-8)
### 3.5 Candidate Pruning Strategy
```python
def prune_candidates(all_users, user_request):
    """
    Multi-stage pruning to reduce candidate pool.
    Each stage returns a subset of the previous stage.
    """
    # Stage 1: Destination
    stage1 = [u for u in all_users if u.destination == user_request.destination]
    
    # Stage 2: Date Window
    stage2 = [u for u in stage1 if has_date_overlap(u, user_request)]
    
    # Stage 3: Budget
    stage3 = [u for u in stage2 if u.budget_tier == user_request.budget_tier]
    
    # Stage 4: Gender Preference (Hard Constraint)
    stage4 = [u for u in stage3 if is_gender_compatible(u, user_request)]
    
    # Stage 5: Age Bucket (Soft - can be relaxed)
    stage5 = [u for u in stage4 if abs(u.age - user_request.age) &lt;= 15]
    
    # Stage 6: Trust Score (Minimum threshold)
    stage6 = [u for u in stage5 if u.trust_score &gt;= 0.4]
    
    # Limit to 500 candidates for performance
    if len(stage6) &gt; 500:
        # Keep users with highest compatibility potential
        stage6 = sorted(stage6, key=lambda u: u.compatibility_potential, reverse=True)[:500]
    
    return stage6
```

### 4.2 Index Strategy Summary
| Table | Index | Type | Purpose |
|-------|-------|------|---------|
| users | phone, email | Unique | Fast lookup by credentials |
| users | verification_status | B-tree | Filter unverified users |
| users | trust_score | B-tree | Filter low-trust users |
| trips | destination, date_range | Composite | Matching pipeline filtering |
| trips | status | B-tree | Filter open trips |
| groups | status, destination | Composite | Group state queries |
| group_members | user_id | B-tree | Find user's groups |
| user_compatibility | score, computed_at | B-tree | Cache invalidation |
| ratings | ratee_id | B-tree | User reputation queries |
| reports | reported_user_id, status | Composite | Risk engine queries |
### 4.3 Partitioning Strategy
```sql
-- Example: Partition trips by month
CREATE TABLE trips_partitioned (
    LIKE trips INCLUDING ALL
) PARTITION BY RANGE (created_at);
-- Create monthly partitions
CREATE TABLE trips_2026_01 PARTITION OF trips_partitioned
    FOR VALUES FROM ('2026-01-01') TO ('2026-02-01');
-- Retention: Drop partitions older than 2 years
DROP TABLE trips_2024_01;
```
### 4.4 Backup &amp; Recovery
| Parameter | Target |
|-----------|--------|
| **RPO** | &lt; 1 hour |
| **RTO** | &lt; 4 hours |
| **Backup Frequency** | Daily full backup + WAL archiving |
| **Retention** | 30 days (daily), 12 months (monthly) |
| **Storage** | AWS S3 with cross-region replication |

## 7. State Machines
### 7.1 Group Lifecycle (Implementation)
```mermaid
stateDiagram-v2
    [*] --&gt; Forming: Group created
    Forming --&gt; Confirmed: Size &gt;= min_size
    Forming --&gt; Archived: Admin override or timeout
    
    Confirmed --&gt; Locked: All members confirm
    Confirmed --&gt; Archived: Admin override or timeout
    
    Locked --&gt; Travelling: Trip start date arrives
    Locked --&gt; Archived: Admin override
    
    Travelling --&gt; Completed: Trip end date passes
    Travelling --&gt; Archived: Admin override or incident
    
    Completed --&gt; Archived: 30 days after completion
    
    Archived --&gt; [*]: Final state
```
**State Transition Rules:**
```javascript
const groupStateTransitions = {
  'forming': {
    allowedTransitions: ['confirmed', 'archived'],
    trigger: {
      'confirmed': (group) =&gt; group.members.length &gt;= group.min_size,
      'archived': (group) =&gt; group.created_at &lt; Date.now() - 7 * 24 * 60 * 60 * 1000 // 7 days timeout
    }
  },
  'confirmed': {
    allowedTransitions: ['locked', 'archived'],
    trigger: {
      'locked': (group) =&gt; group.members.every(m =&gt; m.status === 'accepted'),
      'archived': (group) =&gt; group.confirmed_at &lt; Date.now() - 3 * 24 * 60 * 60 * 1000
    }
  },
  'locked': {
    allowedTransitions: ['travelling', 'archived'],
    trigger: {
      'travelling': (group) =&gt; group.start_date &lt;= new Date(),
      'archived': () =&gt; false // Admin only
    }
  },
  'travelling': {
    allowedTransitions: ['completed', 'archived'],
    trigger: {
      'completed': (group) =&gt; group.end_date &lt; new Date()
    }
  },
  'completed': {
    allowedTransitions: ['archived'],
    trigger: {
      'archived': (group) =&gt; group.completed_at &lt; Date.now() - 30 * 24 * 60 * 60 * 1000
    }
  },
  'archived': {
    allowedTransitions: [] // Terminal state
  }
};
```

## 8. Security Architecture
### 8.1 Authentication Flow
```mermaid
sequenceDiagram
    participant Client
    participant APIGateway
    participant AuthService
    participant UserDB
    
    Client-&gt;&gt;APIGateway: POST /auth/login (email, password)
    APIGateway-&gt;&gt;AuthService: Forward request
    
    AuthService-&gt;&gt;UserDB: SELECT user WHERE email = ?
    UserDB--&gt;&gt;AuthService: User record
    
    AuthService-&gt;&gt;AuthService: Verify password (bcrypt)
    AuthService-&gt;&gt;AuthService: Generate JWT (RS256)
    AuthService--&gt;&gt;APIGateway: JWT + Refresh Token
    
    APIGateway--&gt;&gt;Client: Access Token (expires 24h)
    
    Note over Client,APIGateway: Subsequent requests
    Client-&gt;&gt;APIGateway: GET /users/me (Authorization: Bearer JWT)
    APIGateway-&gt;&gt;APIGateway: Verify JWT signature
    APIGateway-&gt;&gt;AuthService: Forward with user claims
    AuthService--&gt;&gt;APIGateway: User profile
    APIGateway--&gt;&gt;Client: User profile
```

### 8.3 Security Hardening Checklist (Implementation)
| # | Control | Implementation |
|---|---------|----------------|
| 1 | JWT Authentication | RS256 with 2048-bit RSA keys, rotated every 6 months |
| 2 | Password Hashing | bcrypt with cost factor 12 |
| 3 | TLS | TLS 1.3 enforced for all endpoints |
| 4 | Rate Limiting | 100 req/min per IP, 1000 req/min per user |
| 5 | SQL Injection | Parameterized queries (prepared statements) |
| 6 | XSS Protection | CSP headers, output encoding |
| 7 | CSRF | OAuth 2.1 PKCE, SameSite cookies |
| 8 | PII Encryption | AES-256-GCM for PII fields |
| 9 | Audit Logs | All admin actions logged |
| 10 | Secrets Management | AWS Secrets Manager |
| 11 | Input Validation | JSON Schema validation |
| 12 | CORS | Strict origin whitelist |
### 8.4 KYC Integration Architecture
```mermaid
sequenceDiagram
    participant User
    participant Client
    participant AuthService
    participant KYCProvider
    participant UserDB
    participant Kafka
    
    User-&gt;&gt;Client: Submit Aadhaar/DigiLocker
    Client-&gt;&gt;AuthService: POST /users/me/kyc
    
    AuthService-&gt;&gt;AuthService: Generate KYC token
    AuthService-&gt;&gt;KYCProvider: Send verification request
    
    KYCProvider--&gt;&gt;AuthService: Verification URL
    
    AuthService--&gt;&gt;Client: Redirect URL
    
    User-&gt;&gt;Client: Complete verification
    Client-&gt;&gt;KYCProvider: User completes
    
    KYCProvider-&gt;&gt;AuthService: Webhook callback (verified/rejected)
```

## 12. Architecture Decision Records (ADRs)
### ADR-001: Database Selection
**Context:** Need a primary database that supports ACID transactions, complex queries, and JSON data.
**Decision:** PostgreSQL 16
**Alternatives Considered:**
- MongoDB: Flexible schema but lacks ACID transactions for critical flows
- CockroachDB: Distributed but operational complexity
- MySQL: Good but less JSON support
**Rationale:**
- ACID compliance for bookings, payments, KYC
- JSONB for flexible user preferences
- Native partitioning for scalability
- Read replicas for read scaling
- Mature ecosystem and tooling
**Trade-offs:**
- More complex scaling than NoSQL (sharding) 
- Operational overhead of replication
### ADR-002: Event Streaming Platform
**Context:** Need high-throughput event streaming with durable storage for audit.
**Decision:** Apache Kafka 3.x
**Alternatives Considered:**
- RabbitMQ: Good for message queues, less suited for large event logs
- AWS SQS: Serverless but limited ordering capabilities
**Rationale:**
- High throughput and durability
- Exactly-once semantics (idempotent producers)
- Partitioning for ordered events
- Consumer group scaling
- Schema registry for evolution
### ADR-003: Microservices vs Monolith
**Context:** Application with multiple bounded contexts.
**Decision:** Microservices with event-driven communication
**Alternatives Considered:**
- Modular monolith: Simpler but harder to scale
- Serverless: Not suitable for long-running matching jobs
**Rationale:**
- Independent scaling of matching service (computationally intensive)
- Decoupled services for team autonomy
- Event-driven enables async processing
- Fault isolation
### ADR-004: Matching Algorithm Approach
**Context:** Need efficient group formation with hard constraints.
**Decision:** Multi-stage filtering + Constrained Greedy with Local Swap
**Alternatives Considered:**
- Graph-based community detection: More complex, harder to constrain
- ILP/MILP: Exact but NP-hard for large candidate pools
- Genetic algorithms: Unpredictable output quality
**Rationale:**
- Stage 1-4 filtering reduces candidate pool to manageable size
- Greedy ensures fast execution (&lt; 5 min for 10K users)
- Local swap improves quality after initial greedy
- Hard constraints enforced at filtering stage
